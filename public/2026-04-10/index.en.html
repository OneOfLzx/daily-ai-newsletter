USB for Software Developers (19 minute read)

USB Dev Digest

A no-nonsense guide to USB for software folks — April 7, 2026

USB Doesn’t Have to Be Scary
Writing a USB driver for non-standard devices isn’t kernel-level rocket science. It’s more like building a socket app—thanks to userspace tools like libusb.

Key Takeaways

  1. Choose Your Device: Use an Android phone in fastboot mode (easy to find, well-documented, no OS driver interference).
  2. Manual Enumeration:
    • lsusb (Linux) gives VID/PID (18d1:4ee0 for Google’s Nexus/Pixel fastboot).
    • lsusb -t confirms it’s a Vendor Specific Class (no default driver loaded—perfect for experiments).
  3. libusb Magic:
    • Detect devices with hotplug callbacks (matches VID/PID).
    • Talk via Control Endpoint 0x00:
      • GET_STATUS: Tells you if the device is self-powered (yes) or supports remote wakeup (no).
      • GET_DESCRIPTOR: Fetches binary descriptors (device, config, endpoints) — e.g., 2 bulk endpoints (EP1 IN, EP2 OUT).
  4. Windows Workaround: Use Zadig to install Winusb.sys if no driver exists.

Final Thought: You don’t need embedded hardware expertise to use USB—just treat it like a network protocol with endpoints.

How Do We Get Developers to Read the Docs (7 minute read)

Dev Docs Digest

Making docs that actually get read — <current_date>

Today: The secret to docs developers actually read.

Top Story: Two Audiences = One Failed Doc

The author’s "perfect API" had great docs—until they realized: most devs won’t read them. Why? Docs often try to serve two conflicting audiences:

  1. Consumers: Devs using the API want to scan (endpoints, params, examples)—not read internals.
  2. Maintainers: Devs updating the API need deep context (why two calls, legacy tradeoffs).

Mixing both makes docs too long for consumers and unstructured for maintainers.

Key Fixes

  • Layer your docs: Surface-level (consumers) = endpoint name + purpose + example. Collapsible "Details" (maintainers) = internals, history, edge cases.
  • API design first: Intuitive patterns (e.g., /user/orders/1234 → specific order) mean consumers don’t need docs to guess correctly.
  • Skip completeness: Docs should transfer right info to right person—not just check a box.

Quick Win

Next time you write docs: Ask, "Who is this for right now?" If it’s a consumer, cut the "why"—stick to "what" and "how".

The Git Commands I Run Before Reading Any Code (4 minute read)

Codebase Quick Wins

Your 5-minute guide to understanding a repo before opening files — Apr 8, 2026

Skip scrolling first—these git commands reveal a codebase’s hidden pain points in minutes.

Churn Hotspots

git log --format= --name-only --since="1 year ago" | sort | uniq -c | sort -nr | head -20
Find the 20 most-changed files. High churn + no owner = codebase drag (patch-on-patch, unpredictable edits). Cross-reference with bug clusters for top risks.

Bus Factor Check

git shortlog -sn --no-merges
Rank contributors by commits. 60%+ from one person? Red flag. Inactive (6-month gap) or departed? Crisis. Note: Squash merges hide real authors—ask about workflow first.

Bug Clusters

git log -i -E --grep="fix|bug|broken" --name-only --format='' | sort | uniq -c | sort -nr | head -20
Map bug-prone files. Files in both this and churn lists = highest risk (never fully fixed). Depends on clear commit messages.

Velocity Trend

git log --format='%ad' --date=format:'%Y-%m' | sort | uniq -c
Monthly commit counts reveal team momentum: Halved = key member left; steady = healthy; spikes = batching releases.

Firefighting Frequency

git log --oneline --since="1 year ago" | grep -iE 'revert|hotfix|emergency|rollback'
Reverts/rollbacks: A handful = normal; weekly = untrustworthy deploy process (bad tests, missing staging). Zero = stable or vague messages.

These 5 commands take 2 mins to run—they won’t tell you everything, but they tell you which files to read first and what to watch for. No more wandering on day one.

Simplest hash functions (11 minute read)

Hash Bytes

Simplified hash function insights — April 6, 2026

Forget SHA-256/MD5. This week, we’re talking stupid-simple, non-cryptographic hashes for non-adversarial use cases (compilers, vetted data).

The Cheapest Hashes

  • Addition hash: hash = (hash + next_32_bits()) % 2^32. Works for long/unstructured data (±3% bit bias, minimal correlation) but fails for short/structured (domain names: -10% bias, 5k collisions vs SHA-256’s 3k).
  • Rotated addition: Add ((hash <<3) | (hash >>29)) to make it non-commutative (fixes reordering issues).

Fixing Collisions with Foldmul

Addition’s bit flips don’t spread evenly. Foldmul (multiply by a constant, XOR high/low 64-bit halves) reduces collisions to near SHA-256 levels. Best constants: 0xaaa… or 0x555… (lowest error rates).

Hash Table Pro Tips

  • Separate chaining tables don’t need perfect avalanche—just low mod collisions.
  • If using bottom bits for buckets: Mix bits (rotate, CRC). If using top bits: Truncated product works faster.

Final Recipe

  1. Mix with stupid simplicity (addition).
  2. Rearrange bits (foldmul/CRC).
  3. Profit (simple, effective hash for non-critical use cases).

2. Feature Branch Management

  • Separate Repo: Track patches in a Git repo (since Meta’s monorepo lacks branches). For each upstream release, create base branches and feature branches (e.g., debug-tools/7499) that merge forward during upgrades.
  • Benefits: Preserves history, simplifies upstream contributions, and parallelizes work.

Results
