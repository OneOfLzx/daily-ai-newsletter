Different Approaches to Debugging Query Performance in Rails (10 minute read)

Rails Query Performance Brief

Fix slow database queries in Rails — Apr 15, 2026

Key Tools for Debugging

  1. Rails Built-ins:

    • explain (Rails7.1+ adds :analyze for real execution stats)
    • load_async (Rails7+) for parallel independent queries
    • Strict Loading (Rails6.1+) to catch N+1 issues (use n_plus_one_only mode in production)
    • Query Log Tags (Rails7+) to trace query origins

  2. Gems:

    • Bullet (detect N+1/unused eager loads)
    • rack-mini-profiler (browser-side performance badges)
    • PgHero (PostgreSQL dashboard for missing/unused indexes)

  3. DB-Specific:

    • PostgreSQL: pg_stat_statements (track slow queries), auto_explain (log plans)
    • MySQL: Performance Schema, pt-query-digest
    • SQLite (Rails8+): Default WAL mode, EXPLAIN QUERY PLAN

Practical Workflow

  1. Log slow queries (>100ms) via ActiveSupport Notifications
  2. Analyze plans with explain(:analyze)
  3. Check for missing indexes
  4. Optimize: Add indexes, use select/pluck/exists?, batch with find_each
  5. Verify with Benchmark

Quick Wins

  • Enable strict loading in dev
  • Use async_count/async_sum (Rails7.1+) for parallel aggregations
  • Rails8 Solid Cache: Separate DB to avoid cache query competition

Takeaway: Slow queries hurt UX—start with Rails tools, then layer gems/DB insights for fixes.

LLM Reasoning Brief

Top Story: Weak Supervision & LLM Reasoning

New research from UCLA/NYU/Google studies RL for reasoning (RLVR) under three imperfect supervision settings:

  1. Scarce data (as few as 8 examples)
  2. Noisy reward labels
  3. Proxy signals (majority vote/self-certainty)

Key Findings (Quick Bites)

  • Saturation Dynamics Rule:
    • Models with extended pre-saturation phases (Qwen math/science) generalize from 8 examples, tolerate noise, and use proxy rewards.
    • Rapidly saturating models (Llama across all domains) fail all tests.
  • Root Cause: Not lack of diversity (Llama has more) — but unfaithful reasoning: Llama memorizes answers but produces traces that don’t logically support them.
  • The Fix: Combine domain-specific continual pre-training + supervised fine-tuning on explicit reasoning traces before RL. This extends pre-saturation and recovers generalization.

Why This Matters

Weak supervision is common in real-world AI (e.g., limited labeled data). The study shows how to fix models that otherwise fail — critical for building reliable reasoning systems.

Quick Takeaway

For LLMs to reason well with weak supervision: Prioritize reasoning faithfulness over raw diversity, and prep models with domain-specific training + trace fine-tuning before RL.
