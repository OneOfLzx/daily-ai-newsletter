  1. Persistent Context Storage: Captures session events (tool runs, edits, prompts) and processes them into structured observations (titles, narratives, facts) for long-term retention.

  3. Multi-AI Provider Support: Works with Claude, Gemini, and OpenRouter for observation extraction (supports free/paid models).
  4. Privacy & Control:
    • Private Tags: Exclude sensitive content (API keys, personal notes) from storage using <private> tags.
    • Export/Import: Share memories across installations via JSON files.
  5. Cross-Tool Integration: Integrates with Claude Desktop, Cursor IDE, and Gemini CLI.

Technical Stack

  • Languages: TypeScript
  • Runtime: Bun/Node.js
  • Storage: SQLite (with FTS5 for full-text search) + Chroma Vector DB (semantic search)
  • API: Express.js (worker service for processing/querying)

Key Components

  1. Plugin Hooks: Capture session events (e.g., tool usage, file edits) from the host AI tool.
  2. Worker Service: Background service to process observations, manage queues, and serve API requests.
  3. Database Layer: Stores structured observations, session summaries, and user prompts.
  4. MCP Tools: CLI/API tools for searching and retrieving memory (token-efficient workflow).

Use Cases

  • Retrieve past solutions/decisions for a project.
  • Avoid redoing work by leveraging previous context.
  • Share knowledge across team members via export/import.
  • Control what’s stored with private tags for sensitive content.

Decepticon: Autonomous Attack-Defense Platform Summary

Core Vision: The Offensive Vaccine

Decepticon implements an Offensive Vaccine—a continuous hardening loop where:

  1. Offensive Agents (red team) simulate attacks to find vulnerabilities.
  2. Defensive Agents apply automated fixes (patches, WAF rules, config changes).
  3. The loop repeats to strengthen defenses over time, creating a self-healing system.

Key Architecture & Components

1. Dual-Lane Architecture

  • Offensive Lane: Recon, Exploit, Post-Exploit agents (e.g., Nmap, Nuclei, SQLMap).
  • Defensive Lane: WAF, Patch, Config, Detection agents (e.g., ModSecurity rule management).

2. Critical Components

  • Ralph Loop Engine: Orchestrates the continuous cycle (Recon → Attack → Finding → Feedback → Defense → Verify).
  • AttackDefensePlanAdapter: Integrates attack (OPPLAN) and defense plans into a unified workflow.
  • Feedback Pipeline: Converts offensive findings into actionable defensive remediations.
  • Knowledge Graph: Stores attack paths, vulnerabilities, and defensive actions for context-aware decision-making.
  • OpenClaw Integration: Uses OpenClaw's gateway (WebSocket/HTTP), agent runtime, and plugin system for scalability.

3. OPSEC & Security

  • Docker Sandboxes: Isolate agents to prevent real-system impact.
  • Allowlist System: Restrict tool access to safe paths (e.g., /workspace/evidence).
  • SSRF Protection: Mitigate server-side request forgery risks.

How It Works (Continuous Hardening Loop)

  1. Recon: Offensive agents gather target info (subdomains, open ports, services).
  2. Attack: Exploit agents test for vulnerabilities (SQLi, XSS, RCE).
  3. Finding: Extract structured vulnerabilities (CWE, CVSS, MITRE ATT&CK mappings).
  4. Feedback: Convert findings into defensive actions (e.g., "Add WAF rule for SQLi").
  5. Defense: Apply fixes via defensive agents (patch servers, update WAF rules).
  6. Verify: Re-test to confirm vulnerabilities are mitigated.
  7. Convergence: Measure progress with metrics (findings count, immunity rate, severity distribution).

Key Use Cases

  • Continuous Security Testing: Automate red teaming to find vulnerabilities 24/7.
  • Self-Healing Defenses: Fix issues without manual intervention.
  • Purple Teaming: Align red/blue teams via shared knowledge graph and feedback loops.

Core Concepts Recap

Term Definition
Offensive Vaccine Digital vaccine using simulated attacks to strengthen defenses.
Ralph Loop Autonomous cycle of attack → defense → verification.
Finding Structured vulnerability data (CWE, CVSS, MITRE).
RemediationAction Defensive fix derived from a finding (e.g., patch, WAF rule).
AttackDefensePlanAdapter Unifies attack/defense plans for the Ralph Loop.

Things you didn't know about indexes (12 minute read)

DB Deep Dive

Your weekly database performance breakdown — 2026-04-15

Quick, actionable insights on database indexes: what they do, trade-offs, and hidden tools to boost speed.

What Are Indexes?

Think textbook index: sorted entries linking to data rows. Without one, databases do full table scans (check every row). With an index (e.g., Pokémon name), they use binary search—1000x faster for large datasets.

The Big Trade-Off

Indexes speed up reads but slow writes (INSERT/UPDATE/DELETE need index updates). They also take storage and bloat the query planner (more options to weigh). Rule: Don’t index everything.

Common Pitfalls

  • Composite order: An index on (type_1, type_2) works for type_1 or both, but not type_2 alone.
  • Functions break indexes: lower(name) = 'pikachu' ignores an index on name (raw column vs function result).

Hidden Index Gems

  • Functional: Index expressions (e.g., lower(name) for case-insensitive searches).
  • Partial: Only index rows matching a condition (e.g., legendary Pokémon) to save space.
  • Covering: Use INCLUDE to add columns so queries skip the main table (Postgres).

Pro Tip

Run EXPLAIN (or EXPLAIN ANALYZE) to check index usage. Look for Index Scan (good) vs Seq Scan (full table—bad).

Do You Even Need a Database?

A practical breakdown of flat files vs databases for early-stage apps

Databases are just files—so the question isn’t whether to use files (you always do) but which ones: your own or a database’s. For most early-stage apps, the answer might be your own.

We benchmarked 3 flat-file strategies (JSONL) across Go, Bun, Rust, plus SQLite:

  1. Linear scan: O(n) performance (terrible at scale—23 req/s at 1M records).
  2. In-memory map: O(1) reads (Rust leads at ~169k req/s; Bun beats Go here).

When to switch to a DB:

  • Dataset doesn’t fit in RAM.
  • Need to query by non-ID fields (e.g., "find all orders for user X").

The honest take: You probably don’t need a database yet. Flat files with smart strategies work for almost all early-stage apps.

What It Is

iOS 17 introduced RPPairing—a new remote pairing protocol for encrypted IPv6 tunnels to iOS devices, replacing older lockdownd-based pairing for trusted services. It enables access to all RSD (Remote Service Discovery) services over the network.

Core Specs

  • Wire version: 19
  • Crypto stack: Ed25519 (signing), X25519 (key exchange), SRP-3072 (initial pairing), ChaCha20-Poly1305 (AEAD), HKDF-SHA512 (key derivation)

  • RemoteXPC: HTTP/2 + XPC objects (binary data raw)
  • Pairing Data: TLV8 encoding (type-length-value)
  • Device Info: OPack (Apple-specific compact binary format)

  • Pairing File: Plist with Ed25519 keys + UUIDv3 identifier
  • Pair-Verify: Fast path for existing pairs: X25519 DH → HKDF key derivation → Ed25519 signature validation
