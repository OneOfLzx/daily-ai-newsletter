claude-mem captures and stores context from AI assistant sessions (Claude, Gemini, OpenRouter) to inject relevant past information into future interactions. It eliminates "context amnesia" by preserving work history (tool usage, file edits, decisions) and making it searchable.

  • Excludes sensitive content via private tags (<private>) to control what’s stored.

b. Token-Efficient Retrieval

  • 3-layer MCP Workflow:
    1. Search: Find relevant memories (e.g., search(query="auth bug", type="bugfix")).
    2. Timeline: Get chronological context for results (e.g., timeline(ids=[123,456])).
    3. Get Details: Fetch full observation data (e.g., get_observations(ids=[123])).

c. Knowledge Agents

  • Create queryable "brains" from observation history:
    • build_corpus: Compile a slice of history into a corpus.
    • prime_corpus: Inject the corpus into a new session.
    • Query the agent for synthesized, grounded answers.

d. Folder Context

  • Auto-generates CLAUDE.md files per folder (enable via CLAUDE_MEM_FOLDER_CLAUDEMD_ENABLED) to provide directory-level context.

e. Multi-Provider Support

  • Works with Claude, Gemini (free tier available), and OpenRouter (100+ models).
  • Hot-swappable providers (e.g., switch from Claude to Gemini via settings).

3. Installation & Setup

Quick Install

npx claude-mem install

Provider Configuration

Edit ~/.claude-mem/settings.json to set API keys:

{
  "CLAUDE_MEM_PROVIDER": "gemini",
  "CLAUDE_MEM_GEMINI_API_KEY": "YOUR_KEY"
}

IDE Integration

  • Cursor: Use bun run cursor:setup for IDE-specific integration.
  • Claude Desktop: Configure MCP server to enable memory search in desktop sessions.

4. Architecture

Core Components

  1. Plugin Hooks: Capture session events (e.g., tool usage, file edits).
  2. Worker Service: Manages memory processing (Express.js + Bun).
  3. Database Layer:
    • SQLite + FTS5: Full-text search for observations.
    • ChromaDB: Vector search for semantic similarity.
  4. MCP Tools: Expose memory retrieval via a token-efficient API.

  • Worker Status: Check if the worker is running:

  • Stuck Observations: Use bun scripts/check-pending-queue.ts to recover after crashes.
  • Provider Issues: Verify API keys, rate limits, and model selection in settings.

6. Best Practices

  • Token Efficiency: Use the 3-layer MCP workflow to avoid fetching unnecessary data.
  • Corpus Focus: Build small, targeted corpora for knowledge agents to keep context relevant.
  • Private Tags: Mark sensitive content (API keys, personal notes) with <private> to exclude from memory.

Keeping a Postgres queue healthy (20 minute read)

Postgres Queue Health: The Hidden Dead Tuple Problem

By Simeon Griggs (PlanetScale) — April 10, 2026

The Queue Workload Challenge

Postgres queues rely on transient rows (insert → delete) but struggle with mixed workloads (OLTP + analytics) competing for resources. Speed isn’t the issue—cleanup is.

Dead Tuples: Silent Performance Killers

Postgres’ MVCC (Multi-Version Concurrency Control) marks deleted rows as "dead tuples" (not immediately removed). These:

  • Bloat indexes (indexes retain references to dead tuples).
  • Slow scans (executors waste I/O checking invisible dead tuples).

Autovacuum’s Critical Limits

Autovacuum cleans dead tuples but fails when:

  • Active transactions block it: Postgres won’t remove tuples visible to any running transaction (MVCC horizon).
  • Staggered slow queries: Even short (40s) analytics queries, staggered 20s apart, keep the horizon pinned—preventing cleanup indefinitely.

PlanetScale’s Fix: Database Traffic Control™

This tool sets resource budgets for Postgres query traffic to manage mixed workloads. It prioritizes cleanup and stops slow queries from blocking dead tuple removal, keeping queues healthy.

Apple Silicon and Virtual Machines: Beating the 2 VM Limit (12 minute read)

Apple Silicon VM Limit: Bypassing the 2-VM Rule

A deep dive into macOS internals to unlock more virtual machines — [Current Date]

Key Background

Mac Admin Intern Mykola hit a 2-active-macOS-VM limit on Apple Silicon (via UTM), enforced by macOS’s SLA (section 2.B.iii) for dev/test/non-commercial use.

Where the Limit Hides

Initial checks of the Virtualization.framework (userspace) found no hardcoded rule—the restriction lives in Apple Silicon’s XNU kernel (not Intel):

Streaming sort-merge joins in Polars (6 minute read)

Polars Deep Dive: Streaming Sort-Merge Joins

Optimizing large joins with sorted data — Thu, 9 Apr 2026

Key Insight

Joins are often the costliest part of large queries (due to hash table overhead). Polars now supports streaming sort-merge joins for sorted data—cutting memory usage and runtime dramatically.

How It Works

  • Unique keys: Two pointers advance through sorted left/right datasets to find matches (no backtracking).
  • Duplicate keys: A "mark" remembers the start of a matching key run on the right, allowing rewind to pair all left duplicates with the same right run (avoids missing pairs).

When to Use

Polars uses this strategy only if:
✅ Query runs on the streaming engine
✅ Both join keys are known to be sorted (via set_sorted() or upstream sorting)
✅ Join is equality/range on those keys

Caveat: set_sorted() is a trust declaration—if data isn’t actually sorted, results are wrong. Use .sort() if unsure.

Performance Wins

  • Synthetic: 100M rows each (unique keys) → 18x speedup (hash:1.3s → merge:0.07s).
  • Real-world (NYC taxi):38M rows (sorted by timestamp) → 3.5x speedup (hash:0.35s → merge:0.1s).

Quick Tip

Run query.show_graph(plan_stage="physical")—look for "merge-join" in the output.
