垂直代码库（阅读时间：7分钟）

开发架构简报

可扩展前端代码的实用洞察 — [当前日期]

本周深度剖析：放弃「水平」代码库结构，改用垂直分组解决扩展难题。

按类型拆分的问题

大多数项目初期会按技术类型（组件、工具函数、类型）组织代码。但随着项目增长（比如Sentry有10年以上历史、包含200+顶级组件的代码库），这种方式会变得一团糟：

  • useTheme 和 useTodo 放在一起（无逻辑关联）
  • 代码分散（比如 PageFilters 组件在 components/ 下，类型在 types/core/ 下，工具函数在 utils/ 下）

垂直分组的优势

垂直结构按功能（而非类型）组织代码：

  • 示例：src/widgets/ 包含所有与小部件相关的代码（组件、钩子、工具函数、类型）
  • 相关变更的代码集中存放（降低认知负荷）
  • 与功能团队对齐（每个团队负责一个垂直模块，比如「性能分析」团队负责 src/profiling/）

如何落地

  1. 按功能分组：不只是功能/领域——称为「垂直模块」（比如 PageFilters 作为一个垂直模块）
  2. 处理共享代码：用「设计系统」垂直模块存放可复用UI；将共享组件单独作为一个垂直模块
  3. 强制执行边界：使用单体仓库（pnpm工作区）或 eslint-plugin-boundaries 限制私有代码访问

权衡点

  • 确定垂直模块很难（无精确标准）
  • 私有代码存在重复风险（需团队沟通）

但回报显著：一个可扩展、易导航的代码库，对开发者（以及难以处理混乱有机结构的AI代理）都很友好。

网络安全简报

AI与漏洞狩猎洞察 — <当前日期>

AI漏洞狩猎≠工作量证明

Redis创始人antirez驳斥了一个误区：AI网络安全的运作方式并非如工作量证明（PoW）那样——后者算力越大，越能保证找到解决方案。核心区别如下：

  1. 无无限扩展可能：与PoW（算力足够即可保证哈希碰撞）不同，AI漏洞狩猎很快会遇到瓶颈。用于代码分析的大语言模型（LLM）执行会达到饱和（代码状态分支+模型路径）——路径耗尽后，额外的token毫无用处。

  2. 智能>暴力算力：成功取决于模型的智能水平，而非仅GPU数量。弱模型会产生幻觉，识别出相关问题（例如OpenBSD SACK漏洞的窗口验证漏洞），却无法将线索关联起来（溢出+空分支入口）以找到真实漏洞。更强的模型幻觉更少，因此不会误报漏洞（除非它们达到“神话”（Mythos）级别的理解）。

核心结论：未来AI网络安全领域的赢家，不会是GPU最多的企业，而是拥有最佳模型（以及最快获取模型的能力）的企业。

最低放行年龄：被低估的供应链防御手段（阅读需11分钟）

摘要：最低发布时长作为供应链防御手段
Dani Akash认为，设置「软件包最低发布时长（例如7天）」是一种被低估的低门槛防御手段，可有效应对短期恶意供应链攻击。

2026年的Axios供应链入侵事件（攻击者通过被盗令牌发布恶意npm版本，活跃仅4小时）本可被该规则拦截。通过分析2018至2026年间的21起供应链事件，Akash发现：
• 11起被拦截：短期恶意发布攻击（暴露时长不足7天，例如Solana web3.js、Ledger Connect Kit事件）；
• 2起缩小影响：缩短了暴露窗口；
• 对以下情况无效：长期渗透（XZ Utils）、维护者破坏（colors.js）、构建系统攻击（SolarWinds）、CDN问题（Polyfill.io）或合法漏洞（Log4Shell）。

该规则之所以有效，是因为大多数恶意发布攻击属于「速战速决型」（盗取凭证→发布→数小时/数天内清理）。7天延迟可让社区在你安装前发现并移除恶意版本。

工具支持情况：
• JavaScript生态：Bun（秒级）、npm（天级）、pnpm（分钟级）、Yarn4（时长设置）、Deno（需开启标志）；
• Python生态：uv（时长设置）、pip（需手动输入绝对日期，操作繁琐）；
• Rust生态：Cargo实验性标志 + 第三方工具；
• 其他生态（Go、Java、PHP、Ruby）：无支持。

依赖更新机器人（Renovate/Dependabot）支持延迟设置，但安全更新会绕过延迟。

注意事项：并非万能（最适用于定期更新/语义化版本范围；紧急情况可跳过），且不能替代锁定文件/来源验证工具。

核心缺口：半数生态缺乏这一简单防御手段，且已支持工具间存在命名/单位不一致问题（例如Bun用秒、npm用天）。Akash呼吁将默认设置从即时安装改为3-7天延迟，并推动未支持生态的相关功能落地。

最佳安全措施未必复杂——有时只需等待社区先发现威胁即可。

科德克斯入侵三星电视（阅读时间13分钟）

研究人员与OpenAI合作，利用Codex（一款AI模型）将一台旧三星电视（运行过时Chrome和内核的KantS2平台）的权限从浏览器shell提升至root权限。

测试环境包括浏览器立足点、控制器主机（用于构建静态ARMv7二进制文件和托管文件）、tmux驱动的shell监听器、匹配的固件源代码，以及用于绕过三星“未经授权执行防护（UEP）”的memfd封装工具。

Codex的流程包括检查源代码/日志、通过控制器发送命令、读取结果，以及按需构建辅助工具。它发现了关键漏洞：

  1. 浏览器shell可访问的世界可写ntk*设备节点（如ntksys、ntkhdma）；
  2. ntksys驱动在未经验证的情况下接受用户可控的物理地址/大小，并将其映射到用户空间（即“物理映射原语”）；
  3. ntkhdma泄露已知物理地址（DMA缓冲区）以测试该原语。

攻击链步骤：通过ntkhdma泄露DMA缓冲区地址，利用ntksys证明对物理内存的读写权限，扫描RAM（从/proc/cmdline）查找浏览器的凭证（cred）结构（可通过uid=5001识别），将凭证覆盖为零权限，最后启动root shell。

人工指导对Codex保持正确方向至关重要（例如纠正IP混淆、修复电视冻结问题）。该项目表明，成功的关键在于“人工搭建+AI迭代”的协作循环，而非仅靠自动化漏洞挖掘。

把SSH密钥存入TPM芯片！（阅读时间5分钟）

技术安全简报

每周硬件与软件安全小贴士 — <当前日期>

🛡️ 将SSH密钥存储在TPM中（而非文件）

为何重要：大多数现代电脑都配备了TPM（可信平台模块）——用它来保护SSH密钥免受恶意软件、内存及文件系统风险的侵害。

核心要点

  • 优于文件存储：TPM密钥永远不会离开芯片（不像~/.ssh/id_rsa那样存放在磁盘或内存中）。
  • 注意事项：安全性不如便携式HSM（如YubiKey/Nitrokey）（无需物理接触即可访问）；消费级主板在BIOS更新时可能会清空TPM。
  • 设置步骤（精简版）：
    1. 安装tpm2-tools及PKCS#11库（Ubuntu：apt install ...；Arch：pacman -Syu ...）。
    2. 将自己加入tss用户组，注销并重新登录。
    3. 创建PKCS#11存储，使用SO（安全管理员）/用户PIN初始化（注意隐藏PIN，避免出现在Shell历史中！）。
    4. 导入离线生成的SSH密钥（先移除密码短语，转换为PEM格式——务必离线备份！）。
    5. 配置SSH：在.ssh/config中设置PKCS11Provider路径；使用ssh-add -s一次性缓存PIN。

专业建议：避免使用WSL（TPM在其中无法工作）。离线生成密钥以防范ROCA（CVE-2017-15361）等漏洞。

企业正在衡量AI的采用情况，但更应该衡量的是AI能力。（阅读时间：3分钟）

AI采纳率指标误导性强——员工容量才是真实衡量标准

企业常通过追踪AI采纳情况（如登录次数、竞赛参与量）来评估成效，但这忽略了一个关键障碍：员工的工作容量。近期有个例子：某公司为推广AI使用举办礼品卡竞赛，参与度却很低——因为多数员工已处于满负荷状态，额外增加AI的学习曲线、文档工作及任务额外负担只会加重压力，而非提升采纳率。

采纳率指标衡量的是行为（人们是否使用工具？）而非成果（AI是否节省时间或减轻负担？）。多数AI推广项目将AI视为现有工作的“附加项”，但AI在带来回报前需要先投入时间——这正是超负荷团队所欠缺的。

企业应转而衡量以下两点：

  1. 回收时间：实施后60天内，在3项及以上高频、低决策性任务上节省的工时。
  2. 容量余量：员工是否感觉压力减轻（而非仅关注他们是否使用工具）。

成功的AI落地项目会先移除AI可处理的任务，再引入工具；先在负荷较轻的团队中试点，并在60天时衡量成果（而非采纳率）。核心结论：你无法通过激励提升容量，必须通过替代工作（而非增加工作）来创造容量。

为啥AI搞不定前端？（3分钟读完）

为什么AI在前端开发上力不从心

《为什么AI搞不定前端》摘要（nerdy.dev）

AI擅长处理枯燥、通用的前端任务：脚手架搭建、Token映射、基础UI模式都能精准完成——适合快速做无创意的工作。但一旦需要定制化解决方案？它就彻底翻车。

AI的短板

AI做不好这些：

  • 定制交互（滚动动画、微交互）
  • 布局/间距计算（对称性、动态计算）
  • 复杂组件状态
  • 无障碍性（随意使用aria-hidden）
  • 性能优化（卡顿、冗余代码）
  • 适配复杂组件（复杂度越高越慢、越差）

核心原因

  1. 训练数据陈旧：缺现代CSS；依赖过时模板
  2. 无法"可视化"：大语言模型不是渲染引擎——不擅长空间/数学任务（对间距、图标毫无概念）
  3. 不懂"为什么"：不理解架构逻辑（比如状态机、BDD）
  4. 环境适配难：浏览器差异极大（版本、视口、输入方式）——LLM无法精准覆盖

总结

人类是不可预测的目标（偏好、设备随时变），所以AI在定制化前端工作上永远会力不从心。它适合搭脚手架，但针对定制设计，你仍需调整（或从头构建）。

工业时代的遗留影响

1898年弗雷德里克·泰勒提出的「科学管理」（将思考与执行分离），竟成了数字团队的默认模式——尽管开发应用与铲生铁截然不同。如今多数团队仍遵循「规划→设计→生产」流程，设计师与工程师之间频繁交接。

问题不止是沟通不畅

设计产品与最终成品的差异，源于双方视角的根本不同：

  • 设计师关注「感知与叙事」（用户流程、视觉层级）；
  • 工程师聚焦「结构与约束」（数据边界案例、组件稳定性）。

适度张力反能提升质量

若能早期对齐，双方互补的盲区反而能优化产品：

  • 设计师想要的「更轻盈体验」，可能转化为「时序微调」（而非单纯代码减容），既满足感知需求又符合结构逻辑；
  • 工程师的约束也能激发创意解法。

线性流程会破坏协作

  1. 尽早做浏览器原型：以网页为素材（而非空白画布），提前验证约束条件；
  2. 包豪斯式协作：无交接环节——设计师与工程师共同学习、共同开发（参考格罗皮乌斯1919年创办的学校，融合艺术与工艺）。

核心结论：优秀产品的诞生，源于思考与执行无法割裂。

Fly.io 尝试在 Maelstrom 的 SeqKV（支持读取整数、写入及比较并交换操作的顺序一致性键值存储）上构建仅增长计数器，用于包含3个节点、添加/读取请求及分区故障注入的测试。

最初的尝试（单“counter”键的读改写操作）因并发写入导致更新丢失而失败。基于CAS的修复方案（原子循环直至成功）具有非确定性（有时成功，有时失败）。

一种类似无冲突复制数据类型（CRDT）的方法对应G计数器：每个节点写入自身ID对应的键（无冲突），读取时求和所有节点的值。这并非完整的CRDT（缺少合并/比较操作，因SeqKV负责状态同步）。

SeqKV的顺序一致性允许合法异常（例如陈旧读取），但Maelstrom的实现存在特性差异：随机陈旧读取，且仅在实际发生变更时强制返回最新状态。为修复最终的陈旧读取问题（即使经过10秒冷却时间），需在读取前写入一个唯一值（时间戳）——这能确保返回最新状态。

该挑战凸显，仅理解一致性模型并不足够；KV存储的实现细节对获得确定性结果至关重要。

PostgreSQL 多版本并发控制（MVCC）逐字节详解（9分钟阅读）

BoringSQL 字节简报

PostgreSQL 内核解析，简明版 — 2024-10-15

核心要点：PostgreSQL 的 MVCC（多版本并发控制）让读操作永远不阻塞写操作——这一切仅靠每个元组的8字节实现。

头条：MVCC 的8字节秘密

每个元组都携带两个32位事务ID（XID）：

  • t_xmin：插入该元组版本的事务。
  • t_xmax：删除/更新该元组的事务（0表示存活）。

没有「当前版本」表——每个会话通过一个快照（格式：xmin:xmax:xip_list）判断哪些元组可见：

  • xmin：最低活跃事务ID（更早的已处理）。
  • xmax：第一个未分配的事务ID（忽略≥它的所有事务）。
  • xip_list：正在运行的事务（隐藏其变更）。

快速知识点

  • 隔离级别：
    • READ COMMITTED（读已提交）：每条语句使用新快照（事务中途可看到新提交内容）。
    • REPEATABLE READ（可重复读）：BEGIN时固定快照（事务期间数据状态不变）。
  • 死元组：更新操作会给旧版本元组打上t_xmax标记——这些元组会膨胀表空间，直到VACUUM（清理）回收。
  • 长事务：阻塞VACUUM（它们可能仍需要旧元组）。
  • 提示位：对元组的首次SELECT会缓存其提交状态（写入磁盘——这就是SELECT会弄脏页面的原因）。

工具推荐

MVCC 可视化工具：实时测试 MVCC——启动并发会话、切换隔离级别、观察元组 XID 变化，还能看到VACUUM执行过程。

结语

MVCC 不是魔法——它只是8字节 + 快照。想深入了解提示位、冻结等内容，可查看完整文章。

数据偏移 vs 数据漂移（6分钟阅读）

数据漂移 vs 偏移：为何机器学习模型失效（及如何修复）

林赛·麦克唐纳（Lindsay MacDonald）解释了机器学习生产环境中的两个关键挑战：

  • 数据漂移：输入数据的统计特征随时间逐步缓慢变化（如时尚趋势随季节更迭演变）
  • 数据偏移：模型习得的模式出现突然的结构性断裂（如新规落地或全球事件改写既有规则）

两者都会导致训练数据与真实世界输入不匹配，悄无声息地降低模型性能。

检测方法：

  • 漂移：使用统计检验（PSI、Kolmogorov-Smirnov检验）对比滚动窗口内的特征分布与训练数据
  • 偏移：跟踪代理指标（预测置信度、输出分布），同时监控数据质量（缺失值）、Schema变更、流量波动——真实标签往往存在延迟

影响：

  • 漂移：精度逐步累积下降 → 收入损失、预测偏差
  • 偏移：精度断崖式下跌 → 需深度修复（新增特征/建模，而非仅重新训练）

解决方案：数据可观测性（Monte Carlo的核心方向）可提供数据与模型健康状况的端到端可见性，在异常造成损失前自动标记。

字节跳动全新的Seed Agent是工具集成型AI系统，可无人类提示自主调用工具（如代码解释器、计算器等）。核心特性：

  • 工具检索：借助向量数据库，匹配用户查询与相关工具
  • 多步推理：将复杂任务拆解为子任务（如通过中间计算+验证解数学题）
  • 记忆管理：记录过往交互，避免重复调用工具
  • 性能表现：在全新MMLU-Tool基准测试（工具增强型大模型专项测试）中优于GPT-4o
  • 实际应用：已部署于字节跳动内部产品（如抖音智能助手）

2. Mytho：开源角色型智能体

全新开源项目Mytho（GitHub：Mytho-AI/Mytho）推出角色对齐型智能体，内置神话生物角色（如龙负责策略、巫师负责数据分析）。核心特点：

  • 角色一致性：智能体保持与角色匹配的独特行为与语言风格
  • 协作流程：多智能体协同工作（如骑士收集数据、学者分析数据）
  • 自定义功能：用户可创建自定义角色与工具集
  • 社区热度：快速获得超1万GitHub星标；开发者正为Notion、Slack等开发插件

行业背景

这些项目体现向Agent 2.0的转变——从基础文本生成转向：

  • 解决工具过载（高效选工具）
  • 长交互中保持行为连贯
  • 支持开放、可自定义的智能体系统

文章调侃称，已从“养龙虾”（此前比喻过于复杂的AI演示）转向聚焦智能体研究。
